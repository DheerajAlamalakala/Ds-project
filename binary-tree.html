struct Node {
    int data;
    struct Node *left, *right;
};

struct Node* makeNode(int val) {
    struct Node* n = malloc(sizeof(struct Node));
    n->data = val;
    n->left = n->right = NULL;
    return n;
}

// Unlike a BST, placement is a manual
// choice, not a comparison:
struct Node* attach(struct Node* parent,
                     int val, int isLeft) {
    struct Node* n = makeNode(val);
    if (isLeft) parent->left = n;
    else parent->right = n;
    return n;
}

How this works

A plain binary tree only promises that every node has at most two children — there's no rule about where a value "should" go the way a BST enforces with left-smaller, right-larger. That's why this visualizer asks you to pick the parent and side directly: the shape of the tree is a design choice, not a side effect of the values inserted.

This unordered shape is exactly what shows up in expression trees, heaps (which only enforce parent ≥ children, not left vs. right), and decision trees in machine learning — none of which sort by value the way a BST does.

The three traversals — inorder, preorder, postorder — still work exactly the same way on this tree as they did on the BST, since they only depend on the left/right structure, not on any ordering guarantee.

Delete here is intentionally limited to leaf nodes — ones with no children. A node with children can't simply vanish without deciding what happens to its subtree, and since this tree has no value-ordering rule to fall back on (unlike a BST, which can always promote a clear successor), there's no single "correct" replacement. Removing leaves first, working inward, sidesteps that ambiguity entirely.
